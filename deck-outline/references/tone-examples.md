# トーン Before/After 実例

deck-outline の成果物が実際に「AI っぽい」と指摘された表現と、その修正例。Phase 5 の自然な日本語チェックで迷ったら、ここと照らして直す。抽象的な注意書きより、具体例の方が判断が速い。

## 見出し・ラベル（比喩的ジャーゴン → 機能的な言葉）

| Before | After | 理由 |
|---|---|---|
| 背骨（資料全体を貫く一文） | この資料の狙い（一貫性チェック用の軸） | 独自の比喩語より、見出しがそのまま機能を言っている方が自然に読める |
| 本日の問い | アジェンダ | 「問い」を毎回メタ的に強調すると説教くさい。素直に議題として出す |
| ここまでの到達点 | これまでの進捗 | 「到達点」は硬い体言化。話し言葉に近い語で十分 |
| 開示範囲 | 公開範囲 | 契約書・法務寄りの硬い語より、日常語に近い言い方に寄せる |

## 因果チェーン（多段の「だから」 → 平叙文）

Before:
> 案件分割でリポジトリが増える → 細かいルールは形骸化する → 最小限のルール＋仕組みで強制＋役割分担レビュー → 覚えなくても品質が保たれる → だから今日「開示範囲・レビュー分担・Projects×AI」を決める。

After:
> 案件ごとにリポジトリを分けたことで、細かいルールを増やすほど守られなくなるリスクが出てきた。今日は、ルールそのものを最小限にし、AI と git の仕組みで自然に守れる状態を作ることを目指して、公開範囲・レビュー分担・GitHub Projects×AI 活用の3点を決めたい。

「A→B→C→D→だから」と機械的に連鎖させず、1〜2 文の平叙文に畳む。

## 議論ページの構成（寄せ集め → 1 論点＋方針）

Before: 現状／たたき／仕組み／オープン論点が同じページに無秩序に同居し、1 ページで複数の決定を求めていた。

After: 1 論点に絞って方針（たたき、具体を伴う）を示す形に締め、あぶれた要素は【参考】ページかまとめの持ち帰り論点（パーキングロット）へ送る。

## 議論ページの聞き方（毎ページの「問い」欄 → リード文に畳む）

Before: 各【議論】スライドに「問い（この場で決めたいこと）: この進め方でよいか。」という独立ブロックを置いた。数枚続くと「決めたいこと」の欄が繰り返されてしつこく、圧迫感が出る。

After: 独立した問い欄を廃し、リード文の末尾に改行して 1〜2 文で添える。例: 「…実測がそろった段階で仕組みによる強制を判断したいと考えています。（改行）この進め方でよいか、ご意見をいただければと思います。」（2026-07-09 FB）

## 議論ページの中身（選択肢の比較 → 方針の直接提示）

Before（1回目の推敲で実際に書いてしまった例）: 「選択肢: A. 現状通りPMがすべて見る／B（たたき）: メンバーが敵対的セルフレビューを通してから依頼／C. Copilot code reviewも併用する」と3案を並べ、推奨案は「B」とだけ書いてチェック項目は省略した。

After（FBを受けた修正）: 3案の比較をやめ、「方針（たたき）: メンバーが敵対的セルフレビューを通してから依頼し、PMはプロジェクト品質を中心に見る」と直接提示した上で、実際のチェック項目（目的達成・結果目視・ロジック整合性・説明の有無・説明とコード一致・結果の正しさ、等）を具体的に書いた。

**理由**: ①ユーザーが明示的に「比較したい」と言っていないのに選択肢を並べると、意思決定を丸投げされた印象になり圧迫感が出る。②「1ページ1決定」を守ろうとして選択肢を1行の抽象ラベルに削ると、依頼者が既に持っていた具体的なアイデア（チェック項目など）まで失われる。既定は「こちら（推進側）の考えを、判断材料になる具体を伴って直接示す」— 比較は明示的に頼まれたときだけ。
