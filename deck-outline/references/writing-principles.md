# 書き方原則(正本)

資料・スライド・文章の作成/推敲で常時適用する原則。要約版はグローバル `~/.claude/CLAUDE.md` の「書き方原則」節に常設している。**新しい書き方フィードバックを受けたら、本ファイルに経緯付きで追記し、グローバル側の要約も更新する。**

出所: ユーザー本人の推敲の癖を「なぜそう直したか言語化して覚えて」という依頼に基づき抽出したもの。1〜6 は 2026-06-10 の `20260610_copilot-handson-outline-v2.html`、7〜8 は 2026-06-21 の `20260621_copilot-handson-v4-outline.md`、7④⑤⑥・9・10・11 は 2026-07-01〜02 の `20260630_bi-github-operations-deck-outline.md`(BI 部 GitHub 運用ルール議論デック)の推敲・壁打ちから。

## 原則

1. **相手の語彙で書く** — 専門語・カタカナ抽象語より、対象者が普段使う具体語を選ぶ。対象が広いほど徹底する。例:「文脈トレース」→「プロジェクト情報の管理」(解決したい課題感そのものの言葉に寄せる)。ただし相手が既に使う語(例:「コンテキスト管理」)はそのまま可。

2. **抽象・感覚的表現を避け、具体的な行動で書く** — 「〜する感覚を掴む」のような曖昧な到達点は削るか、検証できる行動に落とす。例: ゴールを「実際に使ってみる」「使いどころを 1 つ決める」に。

3. **重複は 1 つに畳む** — 同じ対象を複数ページ・複数工程に分けない。ツール側で自動化・一体化される手順は 1 ステップに統合する。例: テンプレ説明ページの一本化、議事録作成+コンテキスト更新の統合。

4. **議論を増やさない/決めの初期値を置く** — 開いた論点を減らす。決められるものは「こちらの想定」を提示して*確認*に留め、本当に割れる論点だけ*議論*にする。例: 実施形式の独立議論ページを廃し「1 回 90 分で進める想定」をまとめで共有。

5. **ラベルを正確に** — 異質なものを一語で束ねない。確認事項と議論を「問い」で一括しない(「3 つの問い」→「2 点」)。

6. **締めと並走施策を可視化する** — 質疑・フォローアップ・アンケートの時間、進行中の個別パイロットなど、運営面と全体像への配慮を明示する。資料には**「まとめ」ページを必ず置き、「今後」ページは任意**で添える。まとめは本論の振り返り+持ち帰り+個人利用やサポート連絡を 1 枚に畳む(「振り返り」と「まとめ」を別ページに分けない)。

7. **AI っぽさを消す(自然な文章で書く)** —
   - ① リード文を**短文の連打にしない**。細切れの一文を並べず、1 文が 2〜3 行になってもよいので接続詞でつないで自然な流れにする。
   - ② 箇条書きで**体言(単語)を羅列しない**(「掴む/試す/決める」のような単語並べを避け、短い文で書く)。
   - ③ ビジネス資料では**改まった語**を選ぶ(「今日」→「本日」等)。
   - ④ **比喩的な独自ジャーゴンを見出しに使わない**(例:「背骨」→「狙い」、「本日の問い」→「アジェンダ」、「ここまでの到達点」→「これまでの進捗」)。
   - ⑤ **タイトルは体言止め**にし、動詞終わりの一文(「〜を決めていく」「〜に任せる」)で締めない。
   - ⑥ 「だから」の多段因果チェーンを機械的に並べない(1〜2 文の平叙文に畳む)。

8. **ドキュメントは単体で完結させる** — その資料だけで理解できる構成にし、前版(v1・v2 …)との差分や「前回はこうだった」を本文・スライドに持ち込まない。差分追跡が要るなら制作メモ(トレーサビリティ表・設計判断)に留め、読者の目に触れる本文からは外す。

9. **議論ページは 1 ページ 1 決定・既定は方針の直接提示** — 【議論】タグのスライドは 1 論点に絞り、方針(たたき、具体を伴う)をボディで示す。聞きたいことは独立した「問い」「この場で決めたいこと」欄にせず、リード文の末尾に改行して 1〜2 文で添える(2026-07-09 追記: 毎スライドに問い欄が並ぶとしつこいという FB。当初の question-decision 型の独立ブロックを廃止)。現状・たたき・仕組み・オープン論点を無秩序に同居させない。あぶれた要素は【参考】ページかまとめの持ち帰り論点(パーキングロット)へ送る。**選択肢 A/B/C を並べて比較する形は既定では使わない** — ユーザーが明示的に「比較したい」と依頼したときだけ選択肢 2-3・判断基準(3 つまで)を足す。判断に迷う点はこちらの推奨案・方針を直接示して合意を得る(2026-07-02 追記:「複数案の比較より方針と具体案を直接示してほしい」という FB で既定を反転。当初は選択肢 2-3 が既定だった)。「(Yes/No)」と明示的に書くと圧迫感が出るとの FB もあり — 問いかけの文だけで十分、語尾に強制的な選択肢を付け足さない(2026-07-01)。

10. **圧縮のために具体性を落とさない/黙って削らず確認する** — 「1 ページ 1 決定」「簡潔に」を優先するあまり、依頼者が既に示した具体(チェック項目・数値・手順・固有名詞、過去の議事録や決定事項に書かれた中身)を一行の抽象ラベルに縮めない。削るか迷ったら黙って削らず依頼者に確認する。実例: rules-draft §7 の敵対的セルフレビュー 6 項目を「選択肢 B」の一行に圧縮してしまい、「レビュー方針のアイデアがほとんど削れている」と FB を受けた(2026-07-02)。

11. **言われた通りに整えるのではなく、批判的に意見・質問を返す** — 素材(過去の議事録・決定事項・依頼者の発言)に、構成や圧縮の判断に影響する具体的なアイデアがあれば、それを本当に落としてよいか確認する。既存の正式決定(ADR 等)と矛盾する変更をしようとしている場合はそれを明示して依頼者の判断を仰ぐ。ドラフトの中身(技術的な実現可能性、たたきの矛盾点など)も自分で検証してから提示する。「言われた通りに作る」ではなく「より良い資料にするための壁打ち相手」として振る舞う。実例: GitHub Projects の複数リポジトリ集約・複数 Project 同時所属を Web 調査で裏付けてから方針提示した/S5 の「確認したいこと」がたたきと矛盾する形で選択肢を蒸し返していた点を自分で見つけて指摘した(2026-07-02)。

## 適用時の注意

見落としやすいのは (1) 対象者の語彙、(4) 議論ページの乱立、(5) 「問い/点」のラベル、(7) リード文の短文連打・体言羅列・比喩ジャーゴン・タイトル動詞締め、(8) 前版差分の混入、(9) 議論ページで無許可に選択肢比較を持ち込むこと・毎ページに問い欄を置くこと、(10) 具体性の黙った圧縮、(11) 批判的に問い返さず言われた通りに作ってしまうこと。

Before/After の実例は同ディレクトリの `tone-examples.md` を参照。deck-outline の Phase 1・3〜5 には本原則が組み込み済み。批評専用の検査は `deck-critique` skill が担う。
