# Deck Outline — 資料構成を実文言 md で設計する

スライド・提案資料・定例デッキなどを作るとき、**実装（HTML / デザインツール）に入る前に「構成」を実文言の Markdown で固める**ためのスキル。

## なぜ構成を md で分離するのか

スライド作成は、見た目や実装に気を取られると中身の構造がぼやける。さらに HTML を直接書き始めると「構成のレビュー」と「実装の手戻り」が混ざり、章立てや主張のズレを直すコストが跳ね上がる。

このスキルは構成を**実装ツールに依存しない中間成果物**として切り出す：

- 構成 md は人がそのまま読んでレビューできる（タイトル・リード文・ボディが実文言）。意図の言い換え（「〜のねらい」）だけだと、スライドの中身が想像できずレビューにならない。
- 同じ md を html-slide-deck（HTML）にも Claude Design にも渡せる（ツール非依存の引き渡し）。
- 論理の軸を先に通すので、表紙の目的と各ページの主張が繋がる。

**このスキルは実装には進まない。** ゴールは「レビュー済みの構成 md」と「次の実装ステップへの引き渡し」まで。HTML 生成やデザインは html-slide-deck / Claude Design の役割。

## ワークフロー

```
Phase 1 コンテキスト把握＆規約検出
  ↓
Phase 2 資料の狙いを一文で決める
  ↓
Phase 3 章立て・役割設計
  ↓
Phase 4 実文言ドラフト
  ↓
Phase 5 トレーサビリティ＆一貫性点検
  ↓
Phase 6 保存・レビュー gate・引き渡し
```

Phase 2 と Phase 5 を飛ばさない。ここが「目的と中身が繋がらない資料」を防ぐ核心。

### Phase 1: コンテキスト把握＆規約検出

- **素材を読む**: 依頼内容、関連する議事録・フィードバック・決定事項、改訂なら既存版（流用できる部品＝図表・章・言い回しを把握する）。
- **リポジトリ規約を検出する**: その repo の `CLAUDE.md` / `AGENTS.md` から、**出力先ディレクトリ・命名規約・frontmatter スキーマ・lint・git 運用**を読み取る。特定のディレクトリ構造をハードコードせず、その repo の規約に従う（規約が無ければ `docs/` 直下や会話内提示にフォールバック）。
- **読者・目的・ボリューム感・前提知識を掴む**。提示されていなければ対話で引き出す（1 問ずつ。多問を一度に投げない）。
- **言われたことをそのまま実行するのではなく、批判的に意見・質問を返す**。素材（過去の議事録・決定事項・依頼者の発言）に、ページ構成や圧縮の判断に影響する具体的なアイデア・数値・チェック項目があれば、それを本当に落としてよいか確認する。既存の正式決定（ADR 等）と矛盾する変更をしようとしている場合は、それを明示して依頼者の判断を仰ぐ。「言われた通りに整える」ではなく「より良い資料にするための壁打ち相手」として振る舞う。
- **入力メモのインベントリ化**: 依頼者のアイデアメモ・走り書き・会話中の発言を、**項目単位の番号付きリスト**（M1, M2, …）に起こす。フィードバック・決定事項だけでなく、依頼者が書いた素のメモこそ黙って省略されやすい。このインベントリは Phase 5 のトレーサビリティ表の「出所」に必ず含め、全項目の反映先を突合できる状態を作る。

### Phase 2: 資料の狙いを一文で決める

資料全体を貫く主張を **1 文**で書く。表紙の目的と、各ページのユースケース・主張を、この一文にぶら下げる。背景 → 課題 → 解決 → 具体、の因果が一本で通るかを確認する。

狙いが決まらないまま章立てに進まない。「目的（表紙）と具体（各ページ）が繋がっていない」という最頻の指摘は、ここで狙いを先に通すことでしか防げない。

例: 「プロジェクトの品質は“文脈”で決まる。だから対話型 AI でなく文脈を管理できる仕組みを使い、セルフレビューと文脈共有で品質を底上げする。」

狙いを書いたら、その下に因果チェーンを矢印で展開しておく（各章がこの因果のどこに対応するかが後で効く）。この一文・因果チェーンは**制作用のメモ**であり、見出しに比喩的な独自ジャーゴン（「背骨」等）を使わない。「この資料の狙い」のような機能的な見出しで十分（実例は `references/tone-examples.md`）。

### Phase 3: 章立て・役割設計

- 狙いの因果に沿って章を並べる。
- 各スライドに**役割**を与える。意思決定の場の資料（定例・部会・レビュー会）なら 報告／確認／議論／参考 ＋想定時間。説明・ハンズオン資料なら 導入／本論／実践／まとめ 等。
- **【議論】タグのスライドは 1 論点に絞り、聞きたいことはリード文に畳む**: 方針（たたき、具体を伴う）をボディで示し、確認・相談したいことは独立した「問い」「この場で決めたいこと」欄を立てず、**リード文の末尾に改行して 1〜2 文で添える**（html-slide-deck の Pattern 14 と揃える。2026-07-09 追記: 毎スライドに問い欄が並ぶとしつこいという FB を受け、question-decision 型の独立ブロックを廃止した）。現状・たたき・仕組み・オープン論点を無秩序に同居させない — **1 ページ 1 決定**が原則。あぶれた要素は【参考】ページかまとめの持ち帰り論点（パーキングロット）へ送る。
- **既定は方針（たたき）の直接提示。選択肢の並列比較はユーザーが明示的に依頼した場合のみ** — 「A案／B案／C案から選んでください」と丸投げしない。推進側の考えを、判断に迷う点は推奨として添えつつ直接示し、それをもとに議論する。比較が要ると明示された場合だけ、選択肢 2〜3・判断基準（3 つまで）を足す（2026-07-01 追記：初回実運用で「複数案の比較より方針と具体案を直接示してほしい」という FB を受け、既定を反転した）。
- **圧縮のために具体性を落とさない** — 1 ページ 1 決定・簡潔さを優先するあまり、依頼者が既に示した具体（チェック項目・数値・手順・固有名詞）を一行の抽象ラベルに縮めない。方針（たたき）の中身は、読み手がその場で妥当性を判断できるだけの具体を残す。削るかどうか迷ったら黙って削らず、依頼者に確認する。
- **「まとめ」ページは必ず置く。「今後」ページは任意で添える**（説明・提案・ハンズオンなど締めのある資料）。まとめには本論の振り返りと持ち帰り（次に何をするか）を入れる。今後（発展・運用の次の一歩）は、内容が単体で完結していれば省略してよい。**締めページを重複させない** — 「振り返り」と「まとめ」を別ページに分けず 1 枚に畳む。
- 改訂なら各スライドに **🆕 新規 / ✏️ 改訂 / ♻️ 流用** を付け、既存部品の再利用を明示する（改訂版ほど抽象に退行しがち。流用と新規を分けて把握する）。ただしこれは**制作用のメモ**であり、**読者向けの本文・スライドには前版との差分を持ち込まない**（成果物は単体で完結させる。Phase 4 の「単体で完結」を参照）。
- **全体像をテーブルで一覧化**（章・スライド・役割・目安）。ここで枚数とボリュームのバランスを点検する。

### Phase 4: 実文言ドラフト

各スライドを、**そのまま載せる前提の実文言**で書く。意図の言い換えで止めない。最低限：

- **eyebrow**（小見出し・英語ラベル等）
- **タイトル**（体言止め・名詞句を基本に。動詞締めは避ける）
- **リード文**（結論を 1〜2 文、動詞で締めてよい）。**短い文の連打にしない** — 細切れの一文を並べると機械的（AI っぽい）に映る。1 文が 2〜3 行とやや長くなってもよいので、接続詞でつないで自然な流れの文章にする。ビジネス資料では「今日」→「本日」のように改まった語を選ぶ（リポジトリの読者配慮トーンに合わせる）。
- **ボディ**（実際の中身：箇条書きの文言、カードの見出しと項目、表の行、チャット例・コード例・before/after の実テキスト）。**箇条書きで体言（単語）を羅列しない** — 「掴む／試す／決める」のような単語並べは機械的に映る。各項目を短い文で書く（例「議事録作成とクエリレビューを実際に手を動かして試す」）。ラベル＋説明形式のカード見出しは可。
- **図版指示**（**何を置くか**をツール非依存で書く：2 カラム対比 / ステップフロー / before-after / 表 / 概念図 / ツリー。**どう実装するか**＝CSS やコンポーネントは実装スキルに委ねる）
- **バッジ・区分メモ**（任意：役割バッジ、流用元、変更点。**制作メモ扱い**で、読者向け本文には出さない）

抽象スライド（仕組み・概念・アプローチ）には、具体イメージ（コード例・ディレクトリツリー・観点リスト・擬似 UI）を隣接させるか本文に小さく内包する。抽象単体では「で、結局何をどう作るのか」が届かない。

**成果物は単体で完結させる。** その資料だけを読んで理解できる構成にし、前版（v1・v2 …）との差分や「前回はこうだった」を本文・スライドに持ち込まない。改訂の差分追跡が要る場合も、制作メモ（トレーサビリティ表・設計判断）に留め、読者の目に触れる本文からは外す。

### Phase 5: トレーサビリティ＆一貫性点検

- **トレーサビリティ表**: フィードバック・決定事項・要件、および **Phase 1 で作った入力メモインベントリ（M1, M2, …）の全項目**が、どのスライドで反映されたかを表にする（出所 → 内容 → 反映先スライド#）。抜けを可視化する。
- **落とす項目の明示承認**: 反映先が無い項目は黙って落とさず、「意図的に落とす提案」として一覧で依頼者に提示し、承認を得る。承認された項目はトレーサビリティ表に「落とす（承認済み）」と記録する。タイトル・リード文への反映で足りる項目は、その旨を反映先に書く。
- **一貫性チェック（このスキルの肝）**: 各スライドが狙い（Phase 2 の一文）に繋がるかを 1 枚ずつ確認する。表紙の目的と各ユースケース・主張が直結しているか。繋がらないスライドは、主張を書き換える／狙い側を見直す／削る、のいずれかで必ず解消する。**唐突な対比軸を入れない** — 狙いにぶら下がらない切り口（例：品質の話の中に突然出てくる「自分が速くなる」）は、前後の文脈なく置くと浮く。狙いに沿った軸へ言い換えるか、落とす。
- **自然な日本語チェック**: 声に出して読み、実際に打ち合わせでコンサルタントが言う言い回しになっているかを確認する。Phase 4 のタイトル規範（体言止め・動詞締め回避）が全スライドで守れているかを再確認し、比喩的な独自ジャーゴンを見出しに使っていないか（例：「背骨」）、抽象的な体言化で内容が空疎になっていないか（例：「〜という到達点」）、「だから」の多段連鎖で因果を並べていないかを見る。迷ったら `references/tone-examples.md` の Before/After と照らす。
- **プレースホルダ走査**: TBD・空欄・曖昧な要件が残っていないか。残っていれば埋めるか、Phase 7 のオープン論点へ明示的に上げる。

### Phase 6: 保存・レビュー gate・引き渡し

- Phase 1 で検出した規約に沿って md を保存する（壁打ち・設計の成果なら exploration 系、確定構成なら deliverable の前段など、repo の慣習に合わせる）。frontmatter を付け、lint があれば通す。git 運用ルールがあれば従う（ブランチ・コミット確認等）。
- **レビュー gate**: 「構成 md を `<path>` に保存しました。実装前にレビューをお願いします」とユーザーに渡し、**待つ**。修正依頼があれば Phase 3〜5 を回す。
- **引き渡し**: 承認後、次の実装ステップを 2 択で提示する：
  - **HTML 化** → `html-slide-deck` スキル（構成 md を入力に、パターン選択・HTML 実装から始める）
  - **Claude Design** → 同じ md をデザインのブリーフとして渡す
- 実装後の仕上げ確認には `deck-critique` スキル（別の目での批評・推敲）が使える。構成 md とメモインベントリを一緒に渡すと脱落突合まで検査される。

## 出力フォーマット

構成 md の標準構成・実文言の粒度・トレーサビリティ表の書き方は `references/output-template.md` を参照。汎用の最小例も同梱している。自然な日本語チェックで迷ったら `references/tone-examples.md` の Before/After 実例を参照。

## 適用範囲

スライドに限らず、章立てと主張の一貫性が要る資料（提案書・レポート・説明資料）の構成設計に広く使える。実装ツールが HTML でも Claude Design でも PowerPoint でも、構成 md は共通の入力になる。

## このスキルと html-slide-deck の関係

- `deck-outline`（本スキル）＝ **構成**（問いの整理・章立て・実文言 md・一貫性担保）まで。
- `html-slide-deck` ＝ **HTML 実装**（パターン選択・continova デザイン適用・セルフレビュー）。構成 md を入力に受け取って実装する。

「スライドを作りたい」という依頼は、まず本スキルで構成 md を固め、レビュー後に html-slide-deck か Claude Design へ渡す、の順で進める。
