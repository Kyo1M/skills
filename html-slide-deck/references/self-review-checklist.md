# Self-Review Checklist（HTML 実装後の最終チェック）

スライドを書き終えたら、提示する前にブラウザで開きながら以下 7 観点で必ず自己点検する。
1 項目でも引っかかったら直してから提示する（Phase 5）。

過去のレビューで「事後に指摘されて気づいた」種類の不整合をここに集約してある。
特に複数版を iterate する（v2 → v3 → … と改訂する）場面では、観点 1・2・3 を最優先で点検する。
**改訂を重ねるほど抽象寄りに退行しやすい**（再構成で具体例スライドが削られがち）ので、観点 7（抽象 → 具体）も必ず通す。

---

## 1. ストーリーライン整合

スライド単体の出来ではなく、デッキ全体でストーリーが通っているかを確認する。

- [ ] **章間の橋渡し**: 各スライド末尾の bridge-note または次スライドの top-message 冒頭で、論理の接続が明示されているか
- [ ] **並列概念の混在**: 似た構造（n ステップ / n フェーズ / n 軸 など）が連続するスライドで登場する場合、両者の軸の違いが top-message で明示されているか
  - 例: 「S5 の 3 ステップ（学習プロセス）と S6 の 3 フェーズ（組織成熟）は別軸である」を S6 冒頭で 1 文で示す
- [ ] **縦串**: 序盤で導入した主要概念（n 施策 / n 軸 / n 領域）が、後続スライドの該当箇所でタグ付け・対応関係明示されているか
  - 例: S1 で「3 施策」を導入したら、S6 のフェーズ説明の各 li 末尾にも「→ 施策 ①」のような対応タグを付ける
- [ ] **絞り込み・抽出のマッピング**: 上位スライドの n 個を下位スライドで m 個に絞る場合、対応関係（どれが優先選ばれたか）が明示されているか
  - 例: S1 で「3 施策」、S2 で「優先 2 施策」なら、S2 の各カードに「S1 施策 ① に対応」を併記
- [ ] **1 論点 = 1 軸**: 議論論点カードや問いに、性質の違う質問（仕組み論と進め方論、デザイン論と運用論など）を 1 つに詰め込んでいないか
  - 例: 論点 04 で「S4 の 2 軸方針と S5 の 3 ステップ運用」を同時に問うと、議論時にどちらの話か曖昧になる。論点を分割するか、片方に絞る

## 2. コピー一貫性

スライド間・本文と図の間で同じ概念が同じ言葉で呼ばれているかを確認する。

- [ ] **固有名詞・施策名・分類名**の表記が全スライドで完全一致するか
  - 確認方法: 主要キーワードを `grep -nE "<キーワード候補>" <html>` で検索し、揺れがあれば 1 つに統一して全箇所に伝播
  - 例: 「コード生成」「コード生成のスピード」「コード生成スピード」が混在していないか
- [ ] **数値表記**（n 施策 / n 領域 / n フェーズ / n 軸）が冒頭・本文・後続スライドで一致しているか
  - 例: 冒頭で「5 領域」と謳ったら、本文の表の列数・後続スライドの言及も 5 で揃える
- [ ] **表現揺れ**: です・ます / である調、半角/全角、丸数字（①②③）と算用数字（1,2,3）の混在がないか
- [ ] **footer・ヘッダの版情報**: ファイル名・タイトル・版数・日付・ページ番号が全スライドで揃っているか
- [ ] **前版からの修飾表現の削除**: 時間軸（「在籍中の」「中期」「YYYY 年までに」）、スコープ（「BI 部の」「対象を限定して」）、対象範囲などの修飾表現を削除する場合、意図的な判断か単純省略か確認したか
  - 例: 前版で「在籍中の到達目標」だったバッジを v(n) で「到達目標」だけにすると、時間スコープが曖昧になる。意図的削除なら問題なし、単純省略なら復活させる
  - 推奨: 意図的な削除なら理由を `source_prompt` に残す

## 3. 数値・SVG 図解の整合

top-message で謳う数や構造が、ビジュアル要素の中で正しく表現できているかを確認する。

- [ ] **top-message の「n 個」**と、本文要素数（カード数・表の列数・チップ数・SVG 要素数）が一致しているか
- [ ] **集約・分割の理由が読み手に伝わるか**
  - 例: 「5 領域 → 4 段着手順序」のように集約する場合、なぜ集約されたか（並行着手 / 型固有として後置 / 等）が表記されているか
  - 案: top-message 自体を「4 軸 + 型固有観点」のように内訳明示で書く / 集約された要素にラベルを付ける
- [ ] **SVG の矢印・接続線が「全要素から / への流れ」を表現できているか**
  - 例: 3 レーンから 1 つのエンジンへ流れ込む図なら、3 本すべての矢印を描く（1 本だけにしない）
  - チェック方法: SVG を目視で全ノードを順に追い、入出力線がすべて描かれているか確認
- [ ] **SVG キャプションとビジュアルの整合**
  - 例: キャプションが「ループ」「循環」と謳うなら、SVG にも循環矢印を描く / 直線フローならキャプションも「順次」「次回繰り返す」のような表現に揃える
- [ ] **SVG 内ラベルの font-size 階層**: 主要要素のラベル（container / 主役オブジェクト）が、副要素のラベルより小さくないか
  - 例: 「GitHub = 情報の器」のような container ラベルが、内部要素や対比相手の AI エンジンラベルより小さいと階層が崩れる

## 4. 投影前提の可読性

スライドは PC 画面の等倍ではなく、プロジェクタ・Teams 画面共有での縮小表示を前提に確認する。

- [ ] **本文・チェックリスト・注釈帯・コマンド帯が 16px 既定（最低 15px）／ top-message 26px** になっているか（`references/design-system.md`「サイズ階層」の新基準）
  - ツリー・プロンプト等のモノスペース密集要素は 15px まで。補助ラベル・凡例（SVG 図中ラベル含む）の下限は 12px、**11px 以下は使わない**。「パッと見てそのスライドの主張が伝わる」は本文サイズで担保する
  - HTML の `<style>` と inline `font-size` の両方を grep で確認（`grep -oE "font-size:[0-9.]+px" <html> | sort -u`）
- [ ] **はみ出しチェック（はみ出し厳禁・font より優先）**: 文字を大きくした結果、スライドからのはみ出し・`overflow:hidden` でのクリップが無いか。**スライドは固定 1672×941 ＋ scale 方式である前提**（`aspect-ratio` で可変幅にすると狭い窓でクリップする）
  - 機械チェック①（スライド外周）: 各 `.slide` を一時的に `height:auto; overflow:visible` にして自然高さを測り **≤ 941px**（フォント差を見込み 30px ほど余らせる）。固定 1672 幅で測れば窓幅に依存しない（`scale` は見た目だけ縮む）
  - 機械チェック②（内側クリップ）: スライド外周だけでなく、**`overflow:hidden` を持つ内側要素（掲載ボックス・ツリー・グリッドセル）の `scrollHeight ≤ clientHeight`** も確認する（内側でクリップされると外周は溢れず見逃す）
  - はみ出す場合は **4 カラムを 2×2 に組み替える／2 枚に分割** を先に検討 → 次に card padding / line-height を詰める → それでも収まらなければ **その密度高スライドだけ本文を 14px（投影下限 12px まで）へ落として収める**。はみ出したまま提示しない
- [ ] **container / バナー / ラベルの階層**が視覚的に成立しているか
  - 大きい要素のラベルが他要素より小さい不整合がないか（観点 3 の SVG 内 font-size 階層と連動）
- [ ] **線アイコンの一貫性**（アイコンを使う場合）: 全アイコンが共通の作法（同一 viewBox・stroke-width、`<symbol>`+`<use>` で定義、card-head 上は白／白地は Klein Blue）で揃い、**意味とアイコンが 1:1** か
  - 装飾目的の量産になっていないか（本文 callout・badge・箇条書き各行には付けない）。詳細は `references/design-system.md`「線アイコンシステム」
- [ ] **Teams 画面共有・プロジェクタ縮小**を想定し、実機もしくは縮小プレビューで最後列まで読める文字サイズか確認したか
- [ ] **下部 callout への重要情報埋め込み**を避けたか
  - 画面下部の小さい文字は読まれない前提。重要なら top-message / body の主役オブジェクトに格上げ

## 5. frontmatter・出所トレース

派生関係・議事録合意事項が正しく追跡できる状態かを確認する（project-context-template 規約のリポジトリ）。

- [ ] **derived_from と related の区別**: project-context-template 規約（AGENTS.md「frontmatter スキーマ」節）に従っているか
  - `derived_from`: **派生元ファイル**のリポジトリ相対パス（議事録 → 決定/タスク/deliverable のように、本ファイルの直接の上流になった議論記録・過去版・決定）
  - `related`: **横の関連ファイル**（直接派生していないが関連する文書）
  - 例: ある deliverable の元になった議事録・過去版 HTML・参照した decision はすべて `derived_from`。同じトピックの別 deliverable など「横」の関係は `related`
- [ ] **source_prompt に「v(n-1) からの変更点」と「議事録の合意事項のうち反映したもの」を明記したか**
  - 後で「なぜこの版が作られたか」を追跡できるように
- [ ] **議事録マッピング**: 議事録（`docs/minutes/YYYYMMDD_*.md`）の合意事項・持ち越し事項が、本資料のどのスライドにどう反映されたか追跡可能か
  - 推奨: 改訂作業の最後に「議事録の各事項 → スライド該当箇所」のマッピングを source_prompt または別 deliverable として残す

## 6. 自己完結性

スライド単体で議論・運用できる状態かを確認する。

- [ ] **本文・SVG・callout から議事録・タスクファイルパスへの参照が削除されている**か
  - ❌ 「[docs/minutes/...md]」「tasks/...md」「特定 PR/Issue URL」が本文に残っていないか
  - ✓ 出所は frontmatter `derived_from` / `related` にとどめる
- [ ] **前提知識への依存が過剰でないか** — スライドを後から開いた人、社内パスにアクセスできない人、印刷で見る人が読めるか
- [ ] **読者配慮**: 説明対象になっているチームメンバー本人が読んでも、評価的・批判的に響かないトーンになっているか
  - ❌ 「○○ができていない」「△△が大きい」「□□しかやっていない」
  - ✓ 「○○のために△△を用意した」「□□を段階的に提供する」
- [ ] **内部運用メモ**（作成者→説明者向けの注意書き）が資料本文に残っていないか
  - ❌ 「※ 本ページの位置づけ: 詳細図解 — 説明では概要のみ言及」「読み手向けの参考資料」「議論の中心はこのページ」「合意は前ページを中心に取りたい」のような書き手・話し手向けの注釈
  - ✓ 運用情報は frontmatter `source_prompt` か別のプレゼン台本（運用ドキュメント）に逃がす。資料本文は受け手向けに完結させる
  - 理由: 資料は説明者だけでなく、後から開く人・社内パスにアクセスできない人・印刷で見る人・クライアントが読む。受け手にとって意味不明な内部メモは雑音になる
  - 注意: 議事録・decision など **内部運用記録 として作る Markdown** には適用されない。**deliverables/ の HTML スライド（クライアント・部内に共有する資料）** にだけ適用される観点

## 7. 抽象 → 具体の橋渡し

**抽象論で止まったスライドは方針合意には使えても、ターゲット読者に「何をどう作るか」のイメージは届かない**。
仕組み・概念・アプローチを示すスライドは、ターゲット読者の業務に anchor した具体イメージ（コード・ツリー・観点・画面）と必ずペアにする。改訂を重ねるほど抽象寄りに退行しやすいので、改訂版こそこの観点を最優先で。

- [ ] **「仕組み / 概念 / アプローチ」スライドの直後または併置で、具体イメージスライドが用意されているか**
  - 例: 「コードレビュー基盤の仕組み」スライドの隣に「フィードバックループ図 + 観点 skill/prompt スニペット」スライド
  - 例: 「GitHub 集約 × AI」スライドの隣に「リポジトリ構成（ディレクトリツリー）」スライド
- [ ] **各抽象スライド単体で「で、結局何をどう作るのか」が読者に想像できるか**
  - できない場合は、隣に具体イメージスライドを追加する、または抽象スライド内に小さな具体例ボックス（観点 box / コードスニペット小窓）をインライン化する
- [ ] **ターゲット読者の業務に anchor した具体例が混ざっているか**（一般論ではなく、その人の SQL / dbt / 議事録 / KPI 等の言葉で）
  - 「コードレビューが大切」だけでなく「`WITH 句の名前が意図を表しているか` のような観点を skill 化」と書く
- [ ] **改訂版（v2 → v3 → …）で具体イメージスライドを削っていないか**（前版より抽象寄りに退行していないか）
  - 構成変更で具体例ペアが落ちていないかを git diff で確認
- [ ] **「具体イメージが作れない」と判断した抽象スライドは、本当に必要か**
  - 作れないスライドは抽象論で終わる可能性が高い。本当に伝えたいことかを再検討し、削れるなら削る

## チェックリストの運用

- このチェックリストは **HTML 実装直後（Phase 5）に必ず通す**。提示前に 1 項目でも該当があれば修正する
- iterate（v2 → v3 → … の改訂）の場合は、観点 1・2・3 を特に重点的に。前版から構成を変えると章間整合・縦串・表記揺れが崩れやすい
- 議論直後の改訂版を作る場合は観点 5（議事録マッピング）も必ず通す
- 改訂版は観点 7（抽象 → 具体）も必ず通す。再構成で具体例スライドが削られがち
- 投影本番前は観点 4 を実機で確認する

このチェックリストで検出できなかった種類の不整合に気づいたら、本ファイルに項目を追加する。skill 自身が育つ運用にする。
