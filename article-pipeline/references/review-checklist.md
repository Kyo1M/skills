# レビューチェックリスト

記事をクリティカルにレビューする際の観点。1〜4 章は note・Zenn 共通。5 章は note のビジネス記事向け。Zenn の技術記事は技術的正確さ・再現性も併せて確認する（`platform-conventions.md` 参照）。

## 1. 致命的な問題

以下があれば最優先で指摘する。

### タイトルと中身のギャップ

- タイトルが約束していることを記事が果たしているか
- 釣りタイトルになっていないか
- 読者が「思っていたのと違う」と感じないか

### 読者が離脱するポイント

- 冒頭3行で読者を掴めているか
- 導入が長すぎないか（目安：全体の15%以内）
- 結論が最後まで分からない構成になっていないか

### 誤解を招く表現

- 事実と意見が混同されていないか
- 誇張表現がないか
- 専門用語が説明なしに使われていないか

## 2. 構造的な問題

### 導入部のチェック

- 「誰の、どんな課題を、どう解決するか」が明確か
- 共感パートが長すぎないか（読者が既に知っていることを丁寧に説明しすぎていないか）
- 本題に入るまでに何文字かかっているか

### 論理構成のチェック

- 章の順番に必然性があるか
- 論理の飛躍がないか
- 同じことを繰り返していないか
- 各章の役割が明確か

### ターゲットのズレ

- 想定読者のレベルに合っているか
- 事例が読者の業態に合っているか
- 専門性のレベルが一貫しているか

### ボリュームのバランス

- 章ごとの文字数バランスは適切か
- 重要な章に十分な文字数が割かれているか
- 削れる部分はないか

## 3. 言葉の問題

### 刺さらない表現

| パターン | 例 | 改善方向 |
|----------|-----|---------|
| 抽象的すぎる | 「いろいろな課題」 | 具体例を挙げる |
| 主語が曖昧 | 「それが重要です」 | 何が重要か明示 |
| 受動態の多用 | 「考えられています」 | 能動態に変える |
| カタカナ語の乱用 | 「ソリューション」 | 日本語に置き換える |

### 専門用語の扱い

- 初出時に簡潔な説明があるか
- 業界用語を当たり前のように使っていないか
- 略語は正式名称と併記されているか

### トーンの一貫性

- 敬体（です・ます）と常体（だ・である）が混在していないか
- フレンドリーすぎる/堅すぎる箇所がないか
- 読者に寄り添うトーンになっているか

### 説明の厚み（さっぱりしすぎていないか）

- 「なぜそうなのか」の理由が書かれているか
- 抽象的な主張の後に具体例があるか
- 1段落が1〜2文で終わっていないか（目安：3〜5文/段落）
- 読者が「なるほど」と納得できる説明量があるか
- 箇条書きだけで説明を済ませていないか（箇条書きの前後に文脈説明があるか）

## 4. 改善後の構成案テンプレート

レビュー後、以下の形式で改善案を提示する。文字数は**本文のみ**でカウント（画像プロンプトは除く）。

```
| 章 | 内容 | 想定文字数 |
|----|------|-----------|
| 導入 | （概要） | 250字 |
| 1章 | （概要） | 600字 |
| 2章 | （概要） | 700字 |
| ... | ... | ... |
| まとめ | （概要） | 200字 |
| **合計** | | **約2,500〜3,500字** |
```

## 5. 新規事業・MVP開発記事の特有チェック（note ビジネス記事向け）

note でのビジネス・新規事業領域の記事に特化した観点。

### 読者の前提知識

- スタートアップ創業者：実務経験あり、時間がない、具体的なHowを求める
- 大企業の新規事業担当：社内調整が多い、説得材料を求める
- 潜在的な起業志望者：概念理解が目的、入門的な内容を求める

### 信頼性の担保

- 筆者の実体験が入っているか
- 具体的な数字や事例があるか
- 机上の空論になっていないか

### CTAへの導線

- 記事を読んだ後、読者は何をすべきか明確か
- サービスへの導線が自然か（押し売りになっていないか）
- 次の記事への誘導があるか
