# 記事作成パイプラインスキル

note・Zenn 向け記事を、**企画 → 吟味 → 構成 → 執筆 → 推敲 → 画像 → 公開準備**の 7 フェーズで伴走する。

## 対象ディレクトリ

`writing-articles` リポジトリで作業する前提。**1 記事 = 1 フォルダ**。

- `articles/<yyyymmdd>_<slug>/` — 執筆中の記事フォルダ。中に `memo.md` / `article.md` / `assets/` を持つ
- `published/<yyyymmdd>_<slug>/` — 公開済み記事。`articles/` からフォルダごと移動して入れる
- `templates/` — `memo.md` / `article.md` のひな型

```
articles/<yyyymmdd>_<slug>/
├── memo.md        ネタ・着想
├── article.md     記事本文
└── assets/        画像（NN-desc.png）
```

- `<yyyymmdd>` は作成日（執筆開始日）で固定、`<slug>` は kebab-case の英小文字
- ステージは `article.md` の frontmatter `status`（idea / draft / published）で表す
- 詳細な規約は `writing-articles` の `CLAUDE.md` を参照

## 進め方の原則

- **フェーズは独立して呼べる**。「構成だけ」「推敲だけ」「画像だけ」も可。ユーザーの要望から今どのフェーズかを判断する。
- **忖度しない**。良くない部分は明確に指摘し、改善案まで出す。
- **メモを鵜呑みにしない**。ユーザーが渡したメモは出発点であって正解ではない。Phase 2 で批判的に吟味してから構成に渡す。
- **対話的に進める**。確認質問は一度に 5 項目以内。既に明らかなものは省略。
- 体裁ルールはプラットフォーム（note / zenn）で切り替える。詳細は `references/platform-conventions.md`。

---

## Phase 1: 企画

ネタを記事テーマに育てる。

1. `articles/` 配下の各記事フォルダを見渡し、書けそうなテーマを提示する。ネタが特定済みならそれを使う。
2. 新規ネタなら記事フォルダ `articles/<yyyymmdd>_<slug>/`（`<yyyymmdd>` は当日）を作り、`templates/memo.md` を元に `memo.md` を置く。
3. テーマが決まったら **プラットフォーム（note / zenn）** を決める。
   - note: ビジネス・読み物・考え方の言語化向け
   - zenn: 技術記事・実装知見・コード中心向け
4. メモに `platform_candidate` を記入。

## Phase 2: 吟味（メモのクリティカル・レビュー）

メモを「正しい前提」として扱わず、構成へ渡す前に批判的に吟味する。観点の詳細は `references/idea-critique.md`。

1. 吟味はターゲット読者と記事の目的に照らして行う。未確定なら先に確認する。
2. 次の 4 観点でメモを吟味する:
   - **主張の妥当性** — 事実か意見か、過度な一般化・根拠不足・誇張はないか
   - **読者フィット** — 読者の前提知識に合っているか、抱く疑問・反論に答えているか
   - **不足している観点** — 反対意見・代替案・前提背景・具体例・「で、どうすれば」が抜けていないか
   - **切り口（アングル）** — その読者・目的に最も刺さる切り口か。より良いアングルはないか
3. 観点ごとに批評サマリ（気になる点＋なぜ）を提示する。忖度しない。
4. 埋めるべき不足・選ぶべきアングルを、確認質問（5 項目以内・推奨回答つき）にしてユーザーへ渡す。
5. ユーザーの回答を `memo.md` に反映してから構成へ進む。

さらに徹底的に論点を詰めたいときは、`grill-me` スキルを併用してもよい。

## Phase 3: 構成

吟味済みのメモから記事の骨子を作る。

1. 確認質問（5 項目以内、既知のものは省略）:
   - ターゲット読者
   - 記事の目的（読者に何をしてほしいか）
   - 記事の重さ（軽い読み物 / じっくり解説）
   - 単発かシリーズか
2. 見出し構成（H2 / H3）と各セクションの分量の重みをテーブルで提示する。
3. 構成案をユーザーと詰めてから執筆へ進む。

構成テンプレート:

```
| 章 | 内容 | 分量 |
|----|------|------|
| 導入 | （概要） | 軽 |
| 1章 | （概要） | 厚 |
| まとめ | （概要） | 軽 |
```

分量は字数で縛らず「内容が過不足なく伝わる長さ」を狙う（`platform-conventions.md` 参照）。

## Phase 4: 執筆

`templates/article.md` を元に、記事フォルダ内に `article.md` を作る。

- frontmatter を埋める（`title` / `slug` / `platform` / `status` / `created` など）。
- 本文をプラットフォーム別の体裁で書く（`platform-conventions.md`）。
- 「さっぱりしすぎ」を避ける: 主張には理由を、抽象論には具体例を添える。1 段落 3〜5 文。
- 箇条書きと長文を使い分ける（並列の列挙は箇条書き、概念説明・具体例・主張は長文）。

## Phase 5: 推敲

下書きをクリティカルにレビューする。観点の詳細は `references/review-checklist.md`。

提示順:

1. **致命的な問題** — 読者が離脱する / 誤解を招く
2. **構造的な問題** — 導入が長い、論理の飛躍、ターゲットのズレ
3. **言葉の問題** — 刺さらない表現、専門用語の乱用、トーンの不一致
4. **改善後の構成案** — 章立てと分量の重み

タイトル案（複数）と冒頭の引きも点検する。ユーザーが方向性を承認したら改善版を作成する。

## Phase 6: 画像

記事に挿入する画像を生成する。手順の詳細は `references/image-generation.md`。

1. 挿入位置を決め、図解パターンを選ぶ（`references/image-prompts.md` の 10 種）。
2. 各画像の生成プロンプトを作成する。
3. codex CLI で実画像を生成し、記事フォルダ内の `assets/NN-<desc>.png` へ配置する。
4. 本文の挿入位置に `![alt](assets/NN-<desc>.png)` を入れる。
5. codex が使えない場合はフォールバックとして本文に画像プロンプトをインライン記載する。

画像枚数の目安: H2 セクション数のおよそ半分。要点が視覚化される章へ 1 枚ずつ置く。上限 6 枚。

## Phase 7: 公開準備

1. 記事フォルダ `articles/<yyyymmdd>_<slug>/` を `published/` へ**フォルダごと**移動する。
2. frontmatter の `status` を `published`、`published_at` と `url` を記入する。
3. `platform: zenn` の場合は Zenn 形式 frontmatter を生成する（`platform-conventions.md`）。
4. 区切りとして git コミットする。
